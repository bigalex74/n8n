# Руководство по репозиторию

## Структура проекта и организация модулей
Этот репозиторий хранит операционные бэкапы и автоматизацию для стека `n8n-docker`.
- `workflows/`: экспортированные workflow n8n (`*.json`), по одному файлу на workflow.
- `credentials/`: только метаданные credentials (без расшифрованных секретов).
- `system_db_backups/`: таблицы PostgreSQL для восстановления и аудита.
- `infrastructure/`: снимки инфраструктуры (Compose для `n8n` и LightRAG, Caddy, Prometheus).
- `tools/`: служебные скрипты и SQL-утилиты для настройки, проверок и восстановления.
- `telegram-apps/`: FastAPI-сервис и тесты для Telegram-интеграций.
- `docs/`, `PROJECT_DOCUMENTATION.md`, `STRATEGIC_PLAN.md`: архитектурная и процессная документация.

## Сборка, тесты и команды разработки
Запускайте команды из корня (`/home/user/n8n-backups`):
- `./sync_n8n.sh`: основной цикл синхронизации (экспорт workflow, бэкап таблиц, обновление репозитория).
- `./backup_to_yandex.sh`: выгрузка архивов бэкапа в облако.
- `./cleanup_system.sh`: очистка устаревших временных и служебных артефактов.
- `python3 -m pytest telegram-apps/test_api.py telegram-apps/tests/test_n8n_integration.py`: smoke/API тесты.
- `npm --prefix telegram-apps/tests ci && npx --prefix telegram-apps/tests playwright test`: E2E-тесты в браузере.

## Стиль кода и соглашения по именованию
- Python: отступ 4 пробела, `snake_case` для функций и файлов, маленькие модули с одной задачей.
- Shell: совместимость с POSIX/Bash, обязательное экранирование переменных, явная работа с env.
- Имена workflow JSON: сохраняйте формат sanitized-имен (пробелы и спецсимволы заменяются на `_` в `sync_n8n.sh`).
- Не коммитьте секреты, токены и расшифрованные credentials.

## Правила тестирования
- Для каждого изменения поведения добавляйте/обновляйте тест, особенно для sync/restore-сценариев.
- Python-тесты: шаблон `test_*.py`; Playwright-тесты: `*.spec.js`.
- Перед PR для инфраструктурных правок выполните минимум один `./sync_n8n.sh` и релевантный тестовый прогон.

## Коммиты и pull request
- В истории используются авто-коммиты вида `bigalexn8n-<N>: Scheduled auto-backup`; сохраняйте этот формат для плановых бэкапов.
- Для ручных правок используйте короткие осмысленные сообщения с областью изменений (пример: `tools: усилен retry в pgadmin tunnel`).
- В PR указывайте: цель, измененные пути, план отката и доказательство тестов (команда + результат). Скриншоты добавляйте только для UI-изменений в `portal/` или `telegram-apps/static`.

## Безопасность и конфигурация
- Локальные параметры храните в `.env` и `.infisical.json`; не хардкодьте секреты в скриптах.
- Перед синхронизацией проверяйте сетевые предпосылки через `tools/check_proxy.sh`.
