# Резюме

## Что Предлагается

Промышленная система перевода книг (300K+ слов) на базе:
- **n8n** — оркестрация workflow
- **LightRAG** — управление контекстом (глоссарий, summary)
- **Ollama** — локальная LLM (qwen2.5:32b, llama3.2:3b)

## Ключевые Преимущества

| Показатель | Текущее | После внедрения |
|------------|---------|-----------------|
| Качество перевода | Зависит от оператора | Консистентное, score ≥8.5 |
| Скорость (300K слов) | ~10 часов | ~2.5-3 часа |
| Консистентность терминов | Ручная проверка | Автоматический глоссарий (98%+) |
| Мониторинг | Отсутствует | Real-time Grafana dashboard |
| Отказоустойчивость | Ручное вмешательство | Auto-retry, checkpointing, DLQ |

## Стоимость

| Компонент | Стоимость |
|-----------|-----------|
| LightRAG | Бесплатно (open source) |
| Ollama + модели | Бесплатно (локально) |
| n8n | Уже развёрнут |
| PostgreSQL | Уже развёрнут |
| Grafana/Prometheus | Уже развёрнуты |
| **Итого CAPEX** | **0 ₽** |
| **OPEX** | **~500 ₽/мес** (electricity) |

## Ресурсы

- **Время разработки:** 10-14 дней
- **Время тестирования:** 3-4 дня
- **Человеко-часы:** 80-100 часов

# Структурный Анализ (LLM)

## Проблема

Корейские веб-новеллы часто **не имеют явной структуры**:
- Нет заголовков глав
- Нет разделения на арки
- Разделители могут отсутствовать

## Решение: LLM Structure Detection

### Этапы

```
┌─────────────────────────────────────────────────────────────────┐
│           LLM STRUCTURE DETECTION (7-8 минут)                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ЭТАП 0: Regex Pre-Scan (30 сек)                               │
│  ──────────────────────────────────                            │
│  • Поиск явных маркеров: 제, 장, 부, Arc, Chapter              │
│  • Если >10 совпадений → использовать regex                    │
│  • Если <10 → LLM анализ                                       │
│                                                                 │
│  ЭТАП 1: Arc Detection (2 минуты)                              │
│  ──────────────────────────────────                            │
│  • 5 сэмплов: начало, 25%, 50%, 75%, конец                     │
│  • LLM Prompt: "Identify arc boundaries"                       │
│  • Output: JSON с арками (start_offset, end_offset)            │
│                                                                 │
│  ЭТАП 2: Chapter Detection (4 минуты)                          │
│  ──────────────────────────────────                            │
│  • Для каждой арки:                                            │
│  │  - Разбить на чанки (6K слов)                               │
│  │  - LLM: "Is this a chapter boundary?"                       │
│  │  - Batch processing (10 параллельно)                        │
│  │                                                             │
│  • Ищем паттерны:                                              │
│  │  - Time skip: "며칠 후", "몇 시간이 흘러"                  │
│  │  - Location change: "서울로", "현장으로"                   │
│  │  - POV change: "철수의 시점", "영희는"                     │
│  │  - Scene break: ***, ---, пустая строка                     │
│  │  - Narrative pause                                          │
│                                                                 │
│  ЭТАП 3: Title Extraction (1 минута)                           │
│  ──────────────────────────────────                            │
│  • Для каждой границы:                                         │
│  │  - Взять 100 слов после                                     │
│  │  - LLM: "Extract chapter title"                             │
│  │  - Если нет → сгенерировать (auto_generated_title=true)     │
│                                                                 │
│  ЭТАП 4: Validation (30 сек)                                   │
│  ──────────────────────────────────                            │
│  • Проверить размер глав (5K-10K слов)                         │
│  • Если <3K или >15K → flag для review                         │
│  • Проверить арки (все ли содержат главы)                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

# Контроль Качества

## Quality Check Workflow

```
┌─────────────────────────────────────────────────────────────────┐
│                  QUALITY CHECK (LLM)                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Prompt:                                                        │
│  ──────────────────────────────────────────────────────────    │
│  "Evaluate the translation quality from Korean to Russian.     │
│                                                                 │
│  Original:                                                     │
│  {original_text}                                               │
│                                                                 │
│  Translation:                                                  │
│  {translated_text}                                             │
│                                                                 │
│  Rate 1-10 on:                                                 │
│  1. Accuracy (faithfulness to original)                        │
│  2. Fluency (natural sounding Russian)                         │
│  3. Terminology consistency (use glossary)                     │
│  4. Style preservation                                         │
│                                                                 │
│  Return JSON:                                                  │
│  {                                                             │
│    "overall_score": 8.5,                                       │
│    "accuracy_score": 9.0,                                      │
│    "fluency_score": 8.0,                                       │
│    "terminology_score": 9.0,                                   │
│    "style_score": 8.0,                                         │
│    "issues": ["неестественный порядок слов в абзаце 3"],       │
│    "suggestions": ["перефразировать: ..."]                     │
│  }"                                                            │
│                                                                 │
│  Decision:                                                      │
│  ──────────────────────────────────────────────────────────    │
│  • Score >= 8.0: Accept → status='completed'                   │
│  • Score 6.0-7.9: Flag → status='needs_review'                 │
│  • Score < 6.0: Retry → increment retry_count                  │
│                                                                 │
│  Grounding Check (anti-hallucination):                          │
│  ──────────────────────────────────────────────────────────    │
│  LLM: "Does translation contain facts NOT in original?"        │
│  • IF yes → retry translation                                  │
│  • IF no → accept                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

# Отказоустойчивость

## Матрица Таймаутов

| Компонент | Таймаут | Действие | Alert |
|-----------|---------|----------|-------|
| Ollama | 5 мин | Fallback → Pause | ✅ |
| Интернет | 1 мин | Pause | ✅* |
| PostgreSQL | 30 сек | Pause | ✅ |
| LightRAG | 15 сек | Fallback (БД) | ⚠️ |
| Docker | 30 сек | Auto-restart | ❌ |

*Если Telegram недоступен → Alert Queue (отложенно)

## Уровни Защиты

```
┌─────────────────────────────────────────────────────────────────┐
│              FAULT TOLERANCE LAYERS                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. Retry с exponential backoff:                                │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Попытка 1: 00:00 → ❌ → wait 10 сек                     │  │
│  │  Попытка 2: 00:10 → ❌ → wait 30 сек                     │  │
│  │  Попытка 3: 00:40 → ❌ → wait 90 сек                     │  │
│  │  Попытка 4: 01:40 → ❌ → Fail                            │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  2. Circuit Breaker:                                            │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  CLOSED → (5 ошибок за 2 мин) → OPEN → (60 сек) →        │  │
│  │  HALF-OPEN → (тест) → CLOSED/OPEN                        │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  3. Checkpointing:                                              │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  После каждой главы → сохранение в БД                    │  │
│  │  При сбое → возобновление с последней сохранённой        │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  4. Dead Letter Queue:                                          │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  • Автоматический процессор (каждые 5 мин)               │  │
│  │  • Retry failed jobs                                     │  │
│  │  • Alert если >10 записей в DLQ                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  5. Notification Queue:                                         │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  • Если Telegram недоступен → очередь                    │  │
│  │  • Processer: каждые 2 минут                             │  │
│  │  • Max retries: 10                                       │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  6. Fallback стратегии:                                         │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Ollama: qwen2.5:32b → llama3.2:3b → Polza.ai API        │  │
│  │  LightRAG: RAG → direct DB query                         │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
