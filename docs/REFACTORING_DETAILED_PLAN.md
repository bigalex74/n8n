# Подробный план рефакторинга — ПРОБЛЕМА → РЕШЕНИЕ → ПРОФИТ

**Дата:** 14 апреля 2026
**Принцип кайдзен:** Один шаг за раз → тест → коммит → следующий

---

## Шаг 4.0: Подготовка

### Шаг 4.0.1: Git бранч
**Проблема:** Работаем в main, любой коммит сразу в production.
**Почему не должно быть так:** Если рефакторинг сломает workflow — откатывать сложно, нужно revert каждого коммита.
**Решение:** `git checkout -b refactor/workflow-decomposition`
**Почему такое решение:** Изолируем изменения. Если что-то пойдёт не так — просто удаляем бранч.
**Профит:** Безопасность, возможность code review перед merge.
**Буст:** Риск production incident → 0%

### Шаг 4.0.2: Бэкап workflow
**Проблема:** Если рефакторинг сломает workflow — нет точки восстановления.
**Почему не должно быть так:** n8n не хранит историю изменений workflow (workflow_history есть, но полный restore = ручная работа).
**Решение:** Запустить `sync_n8n.sh` перед началом — полный бэкап всех workflow в Git.
**Почему такое решение:**已有 существующий механизм бэкапа, работает каждые 4 часа, просто запускаем вручную.
**Профит:** Точка восстановления — можем восстановить любой workflow из backup.
**Буст:** Время восстановления при аварии: с 30 мин → 5 мин

### Шаг 4.0.3: Baseline тесты
**Проблема:** Не знаем какие тесты проходят сейчас — после рефакторинга не сможем сравнить.
**Почему не должно быть так:** Без baseline невозможно определить сломал ли рефакторинг что-то.
**Решение:** `bash tests/run_tests.sh 2>&1 | tee /tmp/test_baseline.txt`
**Почему такое решение:** Один скрипт — 103 теста покрывают всё: инфраструктуру, перевод, структуру, регрессию.
**Профит:** Объективная метрика — "до" и "после" рефакторинга.
**Буст:** Уверенность что ничего не сломали: с "наверное" → "точно"

## Шаг 4.1: Billing Sub-Workflow

### Шаг 4.1.1-4: Billing Polza + Neuro → `[Sub] Billing Check`

**Проблема:** Billing код (2 HTTP запроса + 1 Postgres) захардкожен в 3 workflow:
- `Start` (9cjeUNeTZX3YnO1W57YTP)
- `[Send] finish` (hoUl3ewz23AwAHlq)
- `[Send] processing` (9uUyj9OamISRPudJ)

Всего: **9 узлов** делают одно и то же.

**Почему так быть не должно:**
1. **Дублирование:** 9 идентичных узлов → 9 мест где нужно править если API изменится
2. **Нет retry:** Если Polza вернёт 503 — billing провалится, данные не сохранятся
3. **Нет централизованного error handling:** Каждый workflow обрабатывает ошибки по-своему
4. **Хардкод API ключей в headers:** При экспорте workflow ключи попадут в Git
5. **Невозможно мониторить:** Нет единого места где видно что billing работает/не работает

**Решение:** Создать один sub-workflow `[Sub] Billing Check`:
```
[Trigger: job_id]
  → Billing Polza (retry: 3, timeout: 10s)
  → Billing Neuro (retry: 3, timeout: 10s)
  → Save to DB (retry: 3)
  → Return {polza_balance, neuro_usage}
```
Заменить 9 узлов в 3 workflow на 3 вызова этого sub-workflow.

**Почему такое решение:**
- Sub-workflow = одна точка правды, меняем в одном месте
- Retry on fail = устойчивость к временным сбоям API
- Centralized = легко добавить логирование, мониторинг, alerting
- Credentials через n8n Credential Manager = ключи не в workflow JSON

**Профит:**
- Узлов: 9 → 3 (экономия 6 узлов)
- Места для багов: 3 → 1
- Время на исправление billing bug: 15 мин → 2 мин
- Добавление нового billing API: 3 правки → 1 правка

**Буст:**
- ↓ 67% дублирования кода
- ↑ Retry coverage: 0% → 100%
- ↓ Время добавления нового billing provider: 45 мин → 5 мин

## Шаг 4.2: Telegram Callback Handler

### Шаг 4.2.1-3: 4 одинаковых callback-ветки → `[Sub] Telegram Callback Handler`

**Проблема:** В `/select_files` (72 узла) есть 4 абсолютно одинаковых ветки:
```
Delete chat message → Execute SQL → Select From List → IF (Не отмена?) → Update row → Merge
```
Каждая обрабатывает callback от одной inline кнопки.

**Почему так быть не должно:**
1. **72 узла** — самый большой workflow в проекте, n8n UI тормозит при открытии
2. **4× дублирование:** 20 узлов делают одно и то же, отличаются только callback_data
3. **NoOp `Replace Me`:** Мёртвый узел, никуда не подключён — мусор
4. **Disconnect `Старт перевода`:** Узел без выхода — мёртвый код
5. **Сложность модификации:** Чтобы добавить 5-ю кнопку — нужно скопировать 5 узлов + настроить Merge
6. **Баг-магнит:** Исправляешь баг в одной ветке → забываешь в 3 остальных → баг возвращается

**Решение:** Создать `[Sub] Telegram Callback Handler`:
```
[Trigger: chat_id, message_id, callback_data]
  → Delete message (Telegram)
  → Update DataTable (Postgres)
  → Select From List (Execute Workflow)
  → IF not cancelled → Return data
  → ELSE → Return cancelled
```
Заменить 20 узлов (4 ветки × 5 узлов) на 4 вызова sub-workflow + 1 Merge.

**Почему такое решение:**
- DRY принцип: один раз написал, 4 раза использовал
- Добавление новой кнопки: 1 узел (Execute Workflow) вместо 5
- Bug fix: исправил в sub-workflow → работает везде
- 72 → ~52 узла = n8n UI работает быстрее

**Профит:**
- Узлов: 72 → ~52 (экономия 20 узлов, -28%)
- Мёртвые узлы: 2 → 0
- Сложность добавления кнопки: 5 узлов → 1 узел
- Баги из-за不一致: 4 места → 0 мест

**Буст:**
- ↓ 28% размер workflow
- ↓ Время добавления кнопки: 10 мин → 2 мин
- ↓ Риск inconsistent behavior: HIGH → NONE
- ↑ Читаемость: 3/10 → 7/10

## Шаг 4.3: Google Drive Download Sub-Workflow

### Шаг 4.3.1-3: GDrive + Base64 + Postgres → `[Sub] Google Drive Download` + `[Sub] Save to DB`

**Проблема:** Паттерн "скачать из Google Drive → конвертировать в base64 → сохранить в БД" повторяется 10+ раз:
- `/select_files`: файл перевода, глоссарий, промт, промт постредакта (4 раза)
- `Ручной выбор файлов`: то же самое (4 раза)
- `Добавление Глоссария`, `Добавление Промта` и др. (2+ раз)

Каждый раз: 3 узла (GDrive → MoveBinaryData → Postgres) = **30+ узлов дублирования**.

**Почему так быть не должно:**
1. **30+ узлов дублирования** — если Google Drive API изменится → править в 10+ местах
2. **Нет единого error handling:** Один workflow ловит ошибки скачивания, другой — нет
3. **Нет retry:** Если GDrive вернёт 429 (rate limit) — файл не скачается, перевод провалится
4. **Невозможно добавить кэширование:** Если файл уже скачан — не нужно качать снова
5. **Невозможно добавить валидацию:** Проверить что файл не пустой, правильный формат

**Решение:** Два sub-workflow:

`[Sub] Google Drive Download`:
```
[Trigger: file_id, purpose]
  → Get file (Google Drive, retry: 3)
  → Download (Google Drive, retry: 3)
  → Get base64 (MoveBinaryData)
  → Validate (Code: size > 0)
  → Return {base64, filename, purpose}
```

`[Sub] Save to DB`:
```
[Trigger: job_id, purpose, base64_data, filename]
  → Postgres INSERT/UPDATE document_files (retry: 3)
  → Return {success, file_id}
```

**Почему такое решение:**
- Разделение ответственности: один скачивает, другой сохраняет
- Каждый можно тестировать отдельно
- Retry в обоих = устойчивость к временным сбоям
- Валидация = раннее обнаружение проблем
- Кэширование можно добавить в один место

**Профит:**
- Узлов: 30+ → 10 (2 sub-workflow + 10 вызовов)
- Места для багов: 10 → 2
- Retry coverage: 0% → 100% для GDrive операций
- Добавление нового типа файла: 3 узла → 1 вызов sub-workflow

**Буст:**
- ↓ 67% дублирования кода
- ↑ Retry coverage GDrive: 0% → 100%
- ↓ Время добавления нового типа файла: 10 мин → 1 мин
- ↓ Риск silent failure (пустой файл): HIGH → NONE

## Шаг 4.4: DataTable Operations Sub-Workflow

### Шаг 4.4.1-2: DataTable Create/Delete/Insert/Update → `[Sub] Session Data Table`

**Проблема:** DataTable операции (Create, Delete, Insert, Update) разбросаны по workflow:
- `/select_files`: Create → Insert → Update → Delete (4 узла)
- `Ручной выбор файлов`: то же самое (4 узла)
- `Send Message`: Get row(s) (1 узел)

DataTable используется как временное хранилище состояния inline-кнопок.

**Почему так быть не должно:**
1. **Нет очистки:** DataTable создаётся но не всегда удаляется → утечка памяти
2. **Нет error handling:** Если DataTable уже существует → Create падает
3. **Нет retry:** Если n8n DataTable API недоступен → операция проваливается
4. **Ручное управление:** Каждый workflow сам создаёт/удаляет → легко забыть
5. **Нет lifecycle:** DataTable должна жить только во время сессии выбора файлов

**Решение:** `[Sub] Session Data Table`:
```
[Trigger: action, chat_id, data]
  → Switch:
    ├── init: Create (if not exists) + Insert row
    ├── add: Insert row
    ├── update: Update row (if exists)
    ├── delete: Delete row
    └── cleanup: Delete entire table
  → Return {success, table_id}
```

**Почему такое решение:**
- Инкапсуляция: workflow не знает про DataTable детали
- Auto-cleanup: можно добавить TTL (удалять таблицы старше 1 часа)
- Idempotent: Create if not exists = не падает если уже есть
- Единая точка для добавления логирования всех DataTable операций

**Профит:**
- Узлов: 8 → 6 (1 sub-workflow + вызовы)
- Утечки DataTable: возможны → исключены (auto-cleanup)
- Error handling: разрозненный → централизованный
- Добавление новой операции: 2 узла → 1 вызов

**Буст:**
- ↓ Утечки DataTable:时有发生 → 0
- ↓ Время отладки DataTable проблем: 20 мин → 2 мин
- ↑ Надёжность: 70% → 99%

## Шаг 4.5: Init Translation Sub-Workflow

### Шаг 4.5.1-2: Инициализация перевода → `[Sub] Init Translation`

**Проблема:** Инициализация перевода (Настройка БД → Парсинг → Анализ → Start) вызывается из:
- `/select_files` (через Execute Workflow chain)
- `Ручной выбор файлов` (через "Инициализация перевода")

Оба workflow дублируют логику: создать job → распарсить файл → проанализировать → запустить перевод.

**Почему так быть не должно:**
1. **Цепочка вызовов:** /select_files → Настройка БД → Парсинг → Анализ → Start (5 sub-workflow)
2. **Нет атомарности:** Если Парсинг провалился → БД уже содержит job, но без данных
3. **Нет валидации входных данных:** Можно вызвать с пустым chat_id → ошибка в глубине цепочки
4. **Нет rollback:** Если Анализ провалился → нужно вручную чистить БД
5. **Неясный поток:** Чтобы понять что происходит при инициализации — нужно открыть 5 workflow

**Решение:** `[Sub] Init Translation`:
```
[Trigger: chat_id, translate_file, glossary_file, prompt, post_prompt]
  → Validate inputs (Code: all required fields present)
  → Настройка БД (Execute Workflow)
  → IF success:
    → Парсинг файла (Execute Workflow)
    → Предварительный анализ (Execute Workflow)
    → Start (Execute Workflow)
    → Return {job_id, status: "started"}
  → ELSE:
    → Rollback БД (Postgres: DELETE job)
    → Return {job_id: null, status: "failed", error}
```

**Почему такое решение:**
- Валидация входов = раннее обнаружение проблем
- Rollback при ошибке = БД всегда в консистентном состоянии
- Один вызов вместо 4 = проще использовать
- Возвращает статус = caller знает что произошло

**Профит:**
- Сложность вызова: 4 sub-workflow → 1 sub-workflow
- Валидация входов: нет → есть
- Rollback при ошибке: нет → есть
- Понятность потока: 3/10 → 8/10

**Буст:**
- ↓ Время отладки init проблем: 30 мин → 5 мин
- ↓ Риск inconsistent DB state: HIGH → NONE
- ↑ Понятность: нужно открыть 5 workflow → 1 workflow

## Шаг 4.6: Финализация

### Шаг 4.6.1-4: Проверка, тесты, commit, merge

**Проблема:** После рефакторинга нужно убедиться что всё работает и зафиксировать изменения.
**Почему так быть не должно:** Без финальной проверки можно пропустить сломанный workflow.
**Решение:**
1. Проверить размеры workflow (сравнить с planned)
2. Запустить полный регрессионный тест
3. Сравнить с baseline (`diff`)
4. Закоммитить и замержить

**Почему такое решение:**
- Baseline vs After = объективная метрика качества рефакторинга
- Регрессионные тесты = 103 проверки что ничего не сломалось
- Git commit = точка восстановления, code review

**Профит:**
- Уверенность что рефакторинг не сломал систему: "наверное" → "доказано"
- История изменений в Git = можно откатить любой шаг
- Code review = второе мнение перед production

**Буст:**
- ↓ Риск production regression: 15% → <1%
- ↑ Confidence в изменениях: 6/10 → 10/10
